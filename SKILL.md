# 日报 — 未补充需求

## 日报文件

```
C:\Users\Administrator\Desktop\报告-{年份}年\日报-{年份}-{月份}月\日报需求记录-{YYYY}-{MM}-{DD}.md
```

当前日期从系统获取。文件不存在则按模板创建。

---

## 表格格式

| 序号 | 日期 | 仓库 | 需求概述 | 涉及模块 | 状态 | 人工工时(h) | AI辅助工时(h) | 备注 |
|------|------|------|---------|---------|------|-------------|---------------|------|

- **序号**：自动递增，解析现有表最后一个序号 +1
- **日期**：格式 MM-DD
- **仓库**：当前工作区所在的 git 仓库名
- **涉及模块**：改动涉及的模块/包路径简述
- **状态**：百分比 — `100%`（已完成）、`0%`（未开始）、`50%`（进行中）
- **备注**：关键数字（文件数、行数变化）、修复的 bug、特殊说明

## 流程

### 第 1 步：汇总当日全部提交 🔴 必须完整，不准截断

```bash
git log --format="%h %ai %s" --since="YYYY-MM-DD 00:00:00" --until="YYYY-MM-DD 23:59:59"
```

**禁止**加 `-n` 限制条数。列出**全部**提交，按时间排序。

同时统计全天变更总量：
```bash
git diff --stat <最早commit>..<最晚commit>
```

输出格式：
```
今日共 N 次提交，时间范围 HH:MM ~ HH:MM，涉及 M 个文件，+XXX/-XXX 行
```

### 第 2 步：合并相似任务 🔴 核心，不准跳过

一条日报记录 ≠ 一次 git commit。**必须**把以下情况合并为一条：

| 合并条件 | 判断方法 |
|---------|---------|
| **同文件反复改** | 多个 commit 改同一个文件 → 合并 |
| **同模块同类改动** | commit 涉及同一个包/模块，且改动性质相似（如都是优化、都是修 bug）→ 合并 |
| **同一功能全链路** | 前端+后端+XML+构建产物，只要属于同一个功能需求 → 合并 |

**合并策略**：

1. **先按模块/功能聚拢**：把所有 commit 按改动领域分组
2. **能合则合**：如果多个分组都属于同一个大方向（如"v3 页面建设""目录日志迁移"），合并为一个顶层需求，内部用 1、2、3…… 编号分述子项
3. **唯一顶层原则**：同一天同一个仓库，优先合并到一个大需求下，子项编号区分具体模块

**合并输出格式**（先展示合并结果，再评估工时）：

```
## 合并结果

### 需求：[一句话概括今天做了什么]

#### 1. [子项名称]
- N 次提交（HH:MM ~ HH:MM）
- 涉及文件：X 个
- 变更量：+XXX/-XXX 行

#### 2. [子项名称]
- N 次提交（HH:MM ~ HH:MM）
- 涉及文件：X 个
- 变更量：+XXX/-XXX 行

#### 3. ...
```

**合并完后问用户**：「以上合并结果是否合理？有无需要拆分或再合并的？」等确认后再进入工时评估。

### 第 4 步：工时评估（按合并后的任务逐条评估）

逐维度展开分析并列出子项估算，**不准直接报总数**，**不准给区间值然后让用户自己选**。

## 工时评估规则

### 维度 1：设计调研

| 子项 | 评估逻辑 | 单耗 |
|------|---------|------|
| 现状摸排 | 需要读几个文件？梳理几种现有模式/写法？ | 每类模式 0.2~0.5h |
| 方案设计 | 需要设计新 API、新类、新方法？几个重载？有无异步/并发/多线程场景要专门设计？每项分开算 | 每项 0.5~1h |
| 依赖追溯 | 要不要追继承链、框架默认行为、配置覆盖？ | 每项 0.3~0.5h |
| 方案对齐 | 需要出方案等用户确认 | 0.5~1h |

---

### 维度 2：代码改动

| 子项 | 评估逻辑 | 单耗 |
|------|---------|------|
| 新增代码 | 新类/新方法/新接口？按复杂度估 | 每项 0.3~1h |
| 机械替换 | 完全相同的模式逐文件替换（IDE 搜索替换可批量完成） | 2~3min/处 |
| 差异修改 | 每处需要单独判断——用哪个重载、参数来源、有无特殊逻辑、是否异步 | 5~10min/处 |
| 上下文切换 | 跨模块越多，加载理解新文件上下文的时间越长 | 每 5 个模块 +0.3h |

**禁止把差异修改当机械替换估**——只要每处需要人工决策，就是差异修改。

---

### 维度 3：测试验证

| 子项 | 评估逻辑 | 单耗 |
|------|---------|------|
| 操作入口 | 列出每个受影响的用户操作按钮/入口，逐个正常流程回归 | 5min/个 |
| 异步场景 | 定时器、邮件、批量任务需要构造触发条件，还要查日志确认 | 12min/个 |
| 关键异常 | 挑关键异常分支、边界条件走一遍 | 5min/个 |
| 日志确认 | 每类操作后查数据库/日志表验证字段正确 | 3min/类 |

**规则**：
- **先枚举操作入口清单**，数清楚再算
- 异步场景 ≠ 看到 ThreadPool 就算——要确认能不能手动触发出日志，不能的话要构造数据

---

### 维度 4：风险返工 🔴 已完成任务用简化公式

| 任务状态 | 公式 | 原因 |
|---------|------|------|
| **100%（已完成）** | `风险返工 = 边界遗漏预估`（只加不减，不乘系数） | 修复 commit 本身就是实际返工，已计入代码改动，不再乘法预估 |
| **50%（进行中）** | `风险返工 =（设计调研 + 代码改动 + 测试验证）× 广度系数 × 深度系数 + 边界遗漏` | 未完成，返工不确定性大，保留乘法 |

**已完成任务的边界遗漏**：仅考虑未覆盖的边界（null、类型转换、兼容性等），不做乘法放大。

**广度系数 / 深度系数**（仅进行中任务使用）：

| 改动广度系数 | |
|------|---------|
| < 5 文件 ×0.2，5~15 文件 ×0.3，15~40 文件 ×0.4，> 40 文件 ×0.5 |

| 影响深度系数 | |
|------|---------|
| 基础设施（全项目依赖）×1.5，核心业务 ×1.3，普通业务 ×1.0，边缘代码 ×0.5 |

---

### 维度 5：沟通协作 🔴 按天算，不按任务算

沟通协作是**整天的共享开销**，不逐任务累加。在最后一条任务评估完后，单独算一笔：

```
全天沟通协作：
- 需求对齐：Xh
- 方案讨论：Xh
- 工时评估：Xh
小计：Xh
```

---

## 人工工时计算公式

```
人工工时 = Σ(每条任务的设计调研 + 代码改动 + 测试验证 + 风险返工) + 全天沟通协作
```

取整到 0.5h。

---

## AI 辅助工时

| 子项 | 评估内容 |
|------|---------|
| 设计协助 | AI 读取代码、梳理模式、出方案、多轮讨论 |
| 代码生成 | AI 生成的代码量（文件数 + 行数） |
| 审查验证 | AI 逐文件 diff 审查、发现 bug、验证逻辑 |
| 修正对话 | 多轮修正、返工、工时讨论 |

```
AI 辅助工时 = 设计协助 + 代码生成 + 审查验证 + 修正对话
```

每项给估算值，最后汇总取整到 0.5h。

## 严禁

- 🔴 **禁止截断提交列表** — `git log` 不准加 `-n`，必须列出当日全部提交
- 🔴 **禁止不合并直接评估** — 同文件/同模块/同功能必须合并后再算
- 🔴 **禁止已完成任务用乘法返工公式** — 已完成的只用边界遗漏
- 🔴 **禁止逐任务累加沟通协作** — 沟通协作按天算，只记一笔
- 🔴 **禁止超 16h 不报警** — 超过必须回检合并是否不足
- 禁止跳过维度直接报总数
- 禁止把差异修改当机械替换估算
- 禁止拍脑袋 2 秒给个数
- 禁止忽略异步/定时器等非直接交互的测试成本
- 禁止 AI 工时只填 1h——如果 AI 参与了多轮设计讨论、代码生成、审查、修正，每一项都要加上

---

> 工时评估不是拍脑袋——先合再算、已完成不乘、沟通共享、上限兜底。少算一个维度就是对自己的不尊重。
