# 日报 — 未补充需求

## 日报文件

```
C:\Users\Administrator\Desktop\报告-{年份}年\日报-{年份}-{月份}月\日报需求记录-{YYYY}-{MM}-{DD}.md
```

当前日期从系统获取。文件不存在则按模板创建。

---

## 表格格式

| 序号 | 日期 | 仓库 | 需求概述 | 涉及模块 | 状态 | 人工工时(h) | AI辅助工时(h) | 备注 |
|------|------|------|---------|---------|------|-------------|---------------|------|

- **序号**：自动递增，解析现有表最后一个序号 +1
- **日期**：格式 MM-DD
- **仓库**：当前工作区所在的 git 仓库名
- **涉及模块**：改动涉及的模块/包路径简述
- **状态**：百分比 — `100%`（已完成）、`0%`（未开始）、`50%`（进行中）
- **备注**：关键数字（文件数、行数变化）、修复的 bug、特殊说明

---

## 流程

### 第 1 步：打开日报文件

根据当前日期找到对应 `.md` 文件。不存在则复制模板创建：

```markdown
# 日报需求记录 — YYYY-MM-DD

**日期：** YYYY-MM-DD
**作者：** 胡志伟

---

| 序号 | 日期 | 仓库 | 需求概述 | 涉及模块 | 状态 | 人工工时(h) | AI辅助工时(h) | 备注 |
|------|------|------|---------|---------|------|-------------|---------------|------|
| 1 | | | | | | | | |
| 2 | | | | | | | | |
| 3 | | | | | | | | |

> 空行为模板，新需求逐行追加。
```

### 第 2 步：从对话上下文提取信息

逐列提取，能推断的直接填，不能的逐项问用户：

- **仓库**：从 git remote / 工作区路径提取
- **需求概述**：从 git diff 变更内容 + commit message 总结，一句话说清做了什么
- **涉及模块**：从变更文件路径提取模块名
- **状态**：已提交推送 = `100%`，代码改完未提交 = `50%`，未开始 = `0%`
- **备注**：`git diff --stat` 汇总的文件数和增删行数，修复的 bug，特殊说明

### 第 3 步：工时评估（🔴 核心，严禁跳过任何维度）

逐维度展开分析并列出子项估算，**不准直接报总数**，**不准给区间值然后让用户自己选**。

---

## 工时评估规则

### 维度 1：设计调研

| 子项 | 评估逻辑 | 单耗 |
|------|---------|------|
| 现状摸排 | 需要读几个文件？梳理几种现有模式/写法？ | 每类模式 0.2~0.5h |
| 方案设计 | 需要设计新 API、新类、新方法？几个重载？有无异步/并发/多线程场景要专门设计？每项分开算 | 每项 0.5~1h |
| 依赖追溯 | 要不要追继承链、框架默认行为、配置覆盖？ | 每项 0.3~0.5h |
| 方案对齐 | 需要出方案等用户确认 | 0.5~1h |

**输出格式**：
```
设计调研：
- 现状摸排：X 种模式 × 0.Xh = Xh
- 方案设计：X 项 × 0.Xh = Xh
- 依赖追溯：X 项 × 0.Xh = Xh
- 方案对齐：Xh
小计：Xh
```

### 维度 2：代码改动

| 子项 | 评估逻辑 | 单耗 |
|------|---------|------|
| 新增代码 | 新类/新方法/新接口？按复杂度估 | 每项 0.3~1h |
| 机械替换 | 完全相同的模式逐文件替换（IDE 搜索替换可批量完成） | 2~3min/处 |
| 差异修改 | 每处需要单独判断——用哪个重载、参数来源、有无特殊逻辑、是否异步 | 5~10min/处 |
| 上下文切换 | 跨模块越多，加载理解新文件上下文的时间越长 | 每 5 个模块 +0.3h |

**禁止把差异修改当机械替换估**——只要每处需要人工决策，就是差异修改。

**输出格式**：
```
代码改动：
- 新增代码：X 项 × 0.Xh = Xh
- 机械替换：X 处 × Xmin = Xh
- 差异修改：X 处 × Xmin = Xh
- 上下文切换：X 个模块 × 0.3h = Xh
小计：Xh
```

### 维度 3：测试验证

| 子项 | 评估逻辑 | 单耗 |
|------|---------|------|
| 操作入口 | 列出每个受影响的用户操作按钮/入口，逐个正常流程回归 | 5min/个 |
| 异步场景 | 定时器、邮件、批量任务需要构造触发条件，还要查日志确认 | 12min/个 |
| 关键异常 | 挑关键异常分支、边界条件走一遍 | 5min/个 |
| 日志确认 | 每类操作后查数据库/日志表验证字段正确 | 3min/类 |

**规则**：
- **先枚举操作入口清单**，数清楚再算
- 异步场景 ≠ 看到 ThreadPool 就算——要确认能不能手动触发出日志，不能的话要构造数据

**输出格式**：
```
测试验证：
- 操作入口：X 个 × 5min = Xh
- 异步场景：X 个 × 12min = Xh
- 关键异常：X 个 × 5min = Xh
- 日志确认：X 类 × 3min = Xh
小计：Xh
```

### 维度 4：风险返工

| 子项 | 评估逻辑 |
|------|---------|
| 改动广度系数 | < 5 文件 ×0.2，5~15 文件 ×0.3，15~40 文件 ×0.4，> 40 文件 ×0.5 |
| 影响深度系数 | 基础设施（全项目依赖）×1.5，核心业务 ×1.3，普通业务 ×1.0，边缘代码 ×0.5 |
| 边界遗漏 | 异步线程上下文丢失、null、字段长度超 DB 限制、旧数据兼容——每个未覆盖的边界 +0.5~1h |
| 修正轮次 | 改 → 测 → 发现 bug → 修 → 再测，预估至少 2 轮 |

**公式**：
```
风险返工 =（设计调研 + 代码改动 + 测试验证）× 广度系数 × 深度系数 + 边界遗漏预估
```

**输出格式**：
```
风险返工：
- （X + X + X）h × 0.X（广度）× X.X（深度）= Xh
- 边界遗漏：X 个 × 0.Xh = Xh
小计：Xh
```

### 维度 5：沟通协作

| 子项 | 评估逻辑 | 单耗 |
|------|---------|------|
| 需求对齐 | 对话中澄清需求和范围 | 0.3~0.5h |
| 方案讨论 | 出方案、等确认、讨论修改 | 0.5~1h |
| 工时评估 | 多轮讨论耗时就属于这一项 | 0.3~0.5h |

**输出格式**：
```
沟通协作：
- 需求对齐：Xh
- 方案讨论：Xh
- 工时评估：Xh
小计：Xh
```

---

## 人工工时计算公式

```
人工工时 = 设计调研 + 代码改动 + 测试验证 + 风险返工 + 沟通协作
```

取整到 0.5h。每个维度输出小计，最后汇总，**不允许跳过任何维度直接报总数**。

---

## AI 辅助工时

| 子项 | 评估内容 |
|------|---------|
| 设计协助 | AI 读取代码、梳理模式、出方案、多轮讨论 |
| 代码生成 | AI 生成的代码量（文件数 + 行数） |
| 审查验证 | AI 逐文件 diff 审查、发现 bug、验证逻辑 |
| 修正对话 | 多轮修正、返工、工时讨论 |

```
AI 辅助工时 = 设计协助 + 代码生成 + 审查验证 + 修正对话
```

每项给估算值，最后汇总取整到 0.5h。

---

### 第 4 步：展示汇总 → 等用户确认

把五个维度的展开分析和汇总数字展示出来，**等用户确认数字**，确认后再写入文件。

### 第 5 步：写入表格

找到第一个空行（各列都是 `| | | | | | | | | |` 或序号大于现有数据行）插入新行。

---

## 严禁

- 禁止跳过维度直接报总数
- 禁止把差异修改当机械替换估算
- 禁止拍脑袋 2 秒给个数
- 禁止低估返工——改的越多出错概率越大
- 禁止忽略异步/定时器等非直接交互的测试成本
- 禁止 AI 工时只填 1h——如果 AI 参与了多轮设计讨论、代码生成、审查、修正，每一项都要加上

---

> 工时评估不是拍脑袋——设计、改动、测试、返工、沟通，五个维度逐一展开，少算一个就是对自己的不尊重。
